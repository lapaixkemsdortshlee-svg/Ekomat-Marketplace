# Audit UX complet — Ekomat (2026-07-18)

> Demande de Thrasher : « si tu étais designer, qu'est-ce que tu améliorerais
> dans toute mon app ? Agis comme un développeur UX professionnel. »
> Audit réalisé sur le vrai code (`index.html`) + la vraie base, dans le
> contexte cible : Android bas/moyen de gamme, données mobiles chères et
> instables (Digicel/Natcom), utilisateurs qui découvrent l'e-commerce, et un
> déficit de confiance à combler (le concurrent réel est WhatsApp).
>
> Méthode : heuristiques standard (Nielsen), parcours par flux, et un biais
> assumé : **tout ce qui touche l'argent et la confiance passe avant le joli**.

## Ce qui est déjà solide (à ne PAS toucher)

L'app a déjà absorbé beaucoup de bonnes décisions UX — les re-triturer serait
du churn : tunnel MonCash guidé 3 étapes avec reprise, familles de fermeture
cohérentes (plein écran vs bottom-sheet), états loading/empty/error avec retry,
passe lisibilité (tailles de police), Kreyòl intégral, Sant Èd data-driven,
stepper escrow avec fil connecteur, cartes profil avec bannière, signature
sonore, parrainage auto-appliqué, chip de rôle, favoris persistés (PR #203),
Tailwind précompilé, SW network-first, avis verrouillés aux vrais achats.

## P1 — À faire avant/pendant le pilote (impact direct sur les transactions)

### 1. Images du feed : aucun `loading="lazy"` (0 occurrence dans le fichier)
Le feed masonry charge TOUTES les images immédiatement. Sur un forfait data
haïtien, c'est de l'argent utilisateur brûlé et un premier rendu lent ; c'est
LE levier perf n°1 côté ressenti.
**Fix** : `loading="lazy" decoding="async"` sur les `<img>` des cartes
(feed, similaires, boutique, favoris) — pas sur l'image héro de la fiche
produit. Prévoir un `aspect-ratio`/hauteur fixe (déjà le cas : conteneurs à
hauteur fixe) pour éviter les sauts de layout. **Effort : S.**

### 2. `confirm()` natif côté acheteur pour annuler une commande (ligne ~11781)
« Ou si ou vle anile komand sa a? » s'affiche dans la boîte système du
navigateur : boutons **OK/Annuler en anglais/français système**, look qui casse
la confiance au moment le plus sensible (décision d'argent). L'app possède déjà
`appConfirm()` stylé (utilisé ailleurs).
**Fix** : remplacer par `appConfirm`. **Effort : S.**

### 3. Rassurance anti-double-paiement dans le tunnel MonCash
Scénario réel du pilote : l'acheteur envoie l'argent MonCash, la connexion
meurt avant de soumettre la référence. Au retour, la peur naturelle = « èske m
dwe peye ankò ? » → risque de **double envoi réel**. Le backend est idempotent
(re-soumettre la même transition est un no-op), mais rien ne le DIT à
l'utilisateur.
**Fix** : dans l'étape 2 du tunnel (Referans), une ligne permanente :
« ⚠️ Ou deja voye lajan an? PA voye l yon dezyèm fwa — antre referans lan,
li se prèv ou. » + à la reprise du tunnel (déjà gérée), un rappel explicite
que le paiement précédent est retrouvé. **Effort : S.**

### 4. Orthographe incohérente du mot le plus fréquent de l'app
`kòmand` (57×) vs `komand` (39×) — parfois dans le même écran. Pour un public
qui lit le Kreyòl à des niveaux variés, l'inconsistance ressemble à de la
négligence et nuit à la crédibilité.
**Fix** : normaliser sur `kòmand` (orthographe standard) partout dans les
chaînes UI (pas les identifiants techniques). **Effort : S (sed prudent +
relecture).**

## P2 — Confiance & conversion (après le pilote, avant l'acquisition)

### 5. Avatar/bannière vendeur : stockage local seulement (dette connue #168)
`uploadProfilePhoto`/`uploadCoverPhoto` écrivent un data-URL dans
`localStorage` : pas cross-device, et surtout **invisibles pour les
acheteurs**. Or la photo du vendeur est un des premiers signaux de confiance
d'une boutique.
**Fix** : bucket Storage `avatars` + colonnes `profiles.avatar_url` (déjà
lue partout) / `cover_url`, upload compressé comme les photos produit (le
pattern existe : `product-images`). **Effort : M.** C'est le chantier confiance
n°1 post-pilote.

### 6. Dialogues admin natifs (`prompt()` ×3, `confirm()` ×1)
Rezon rejè, rezon ranbousman, nòt litij, konfimasyon « Lage lajan » passent par
`prompt()/confirm()` système. Admin = Thrasher seul aujourd'hui, donc pas
bloquant — mais « Lage X HTG bay vandè » est l'action la plus irréversible de
l'app dans la boîte la plus moche.
**Fix** : petite sheet de confirmation réutilisable avec champ note (le pattern
`adminConfirmPaymentSheet` existe déjà). **Effort : M.**

### 7. Zones claires en mode sombre dans la feuille carte/localisation
`#mapSheet` garde des fonds inline clairs (`#f6f3ee`, `#fff` pour les
suggestions) qui restent clairs en mode sombre → patchwork. D'autres sheets
récents sont propres.
**Fix** : passer ces fonds en tokens `--dm-*` ou classes surchargées dark.
**Effort : S-M.**

### 8. Recherche produits limitée au cache mémoire
`doSearch` filtre `PRODUCTS` (les ~100 derniers chargés). Avec un catalogue
qui grossit, un produit existant mais non chargé devient introuvable — la
recherche vendeurs, elle, interroge déjà le serveur (RPC `search_sellers`).
**Fix** : requête Supabase `ilike`/trigram sur `products.title` (fallback
local hors ligne). **Effort : M.** Pas urgent avec 1 produit actif ; bloquant
dès ~200 produits.

## P3 — Polish & dettes assumées (plus tard, noté pour ne pas oublier)

- **Notes vocales en base64 dans `messages.content`** : lourd en data (envoi ET
  chargement du fil) — migrer vers Storage un jour ; choix single-file assumé.
- **`completed` jamais atteint** : la commande termine à `released` même après
  l'avis (déjà noté dans l'audit chemin-critique) — avancer à `completed` après
  notation, purement cosmétique.
- **`submitRating` insère sa notification en direct** au lieu de passer par
  `notify()` (convention #5).
- **Focus/scroll après fermeture des sheets** : le focus ne revient pas à
  l'élément déclencheur (a11y clavier/lecteur d'écran — audience minime sur la
  cible, à traiter avec le build natif).
- **Deep-links produit** (`#product=…`) : OK techniquement ; ajouter un bouton
  « Pataje » visible sur la fiche produit pour nourrir la boucle WhatsApp
  (aujourd'hui le partage passe surtout par le parrainage).

## Ce que je recommande de NE PAS faire maintenant

- **Refonte visuelle globale** : l'identité (cream/teal/rust, Manrope/Jakarta)
  est cohérente et distinctive. Le verdict du 16 juillet tient.
- **Feed « mode découverte »** : déjà parké par décision explicite, réévaluer
  avec de vrais vendeurs.
- **Onboarding plus riche** : coach-marks + carte de bienvenue suffisent ;
  chaque écran d'onboarding en plus coûte des installs sur la cible.

## Séquence proposée

1. **Lot pilote (P1, 1 PR)** : lazy-loading images + appConfirm annulation +
   rassurance anti-double-paiement + normalisation `kòmand`. Petit, sûr,
   direct pour le pilote.
2. **Lot confiance (P2, post-pilote)** : avatars/bannières publics (Storage),
   puis dialogues admin, dark mode mapSheet.
3. **Lot échelle (P2/P3, quand le catalogue grossit)** : recherche produits
   serveur, pagination feed, notes vocales Storage.
